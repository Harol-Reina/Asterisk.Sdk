# Product opportunities for v1.12.0 — three-angle audit

A deep investigation run on 2026-04-19 to answer a single question: given v1.11.1 shipped, what is the right scope for v1.12.0 if we optimize for the final product rather than the easiest release? The investigation used three parallel exploration passes — one on the internal codebase for real gaps, one on archived research/plans for parked work, one on the external market for competitive signal — and the findings converged on a coherent strategic direction that is different from the housekeeping release that was the initial proposal.

- **Date:** 2026-04-19
- **Status:** Research — final
- **Related docs:** [2026-04-19-product-alignment-audit.md](./2026-04-19-product-alignment-audit.md) · [../plans/active/](../plans/active/) · [../decisions/](../decisions/)

## §1 Investigation method

Three independent parallel investigations were run with explicit instructions to avoid shortcuts and focus on product value:

1. **Internal codebase audit.** Full scan of `src/`, `Tests/`, `Examples/` for TODOs, FIXMEs, skipped tests, dead interfaces, ghost options, `[Obsolete]` members, public APIs with single implementations, missing extensibility points, and stale documentation. 12-category report.
2. **Deferred work audit.** Full scan of `docs/research/archived/`, `docs/plans/archived/`, `docs/plans/completed/` (including recently migrated superpowers content), `docs/specs/archived/`, `docs/decisions/`, git log, memory files, and the sibling `Asterisk.Sdk.Pro` repo. Classified each finding as parked, partial, abandoned, or superseded.
3. **External market audit.** WebSearch + WebFetch on asterisk-java, AsterNET, Asterisk 22/23/24 release notes, community.asterisk.org, GitHub issues across the Asterisk ecosystem, VoiceAi provider landscape 2026, OpenTelemetry semantic-convention trends, multi-node deployment patterns, and message-broker integration.

The three passes were designed to be independent — no cross-contamination of findings — so that convergence between them constitutes real signal.

## §2 Key findings by angle

### Internal codebase audit

The SDK is in exceptional shape. Zero TODOs, zero FIXMEs, zero skipped tests apart from one environmental ARI container skip. Every `*Options.cs` property is used. Every public interface has either the implementations the ADRs describe or is an explicit extension point for consumers. No `[Obsolete]` members. Every `PublicAPI.Unshipped.txt` is empty of real pending APIs.

Two low-impact findings worth noting:

- `ITopicPermissionMap` is defined in `Asterisk.Sdk.Push.Authz` and appears in `PublicAPI.Shipped.txt` but has zero call sites in the SDK itself. It is a consumer-facing extension point; consumers implement it to map permissions to topic patterns. Not dead code in the strict sense — the design is "consumer provides the map" — but worth a doc note.
- The `Asterisk.Sdk.Push` README says "coming v1.7" for a Pro-SDK feature. Stale as of v1.11.1; update to "available in Asterisk.Sdk.Pro" or drop the timing.

Neither constitutes a product gap that justifies a release on its own.

### Deferred work audit

The interesting finding here is that the `docs/plans/completed/` directory contains several files migrated from the former `docs/superpowers/` folder during the 2026-04-19 cleanup (commit `d663ffc`). Some of those files are plans that were **written but never executed** — they live in `completed/` because the migration preserved the structure of the source tree, not because their work shipped. Three of them are directly relevant to v1.12.0:

- [plan-chan-websocket-integration.md](../plans/completed/plan-chan-websocket-integration.md) — 48 KB, dated 2026-03-04. Full design for a `chan_websocket` audio streaming client including architectural decisions (TcpListener + `WebSocket.CreateFromStream()` for ~40 LOC handshake, AudioSocket primary + WebSocket secondary behind a common `IAudioStream` seam), level-by-level implementation breakdown from ARI REST completeness through typed event dispatch to the audio streaming server and activities integration. Never executed.
- [plan-blazor-dashboard-example.md](../plans/completed/plan-blazor-dashboard-example.md) — 22 KB. Blazor Server dashboard with live queue status, call history, agent metrics, recordings browser. Partially implemented in `Examples/DashboardExample/` (exists but minimal). Belongs more to the `PbxAdmin` repo now that it has been separated.
- [plan-campaign-metrics.md](../plans/completed/plan-campaign-metrics.md) — 16 KB. Outbound dialer with campaign management, dial rate, completion rate, SLA compliance. Commercial territory — fits the Pro SDK, not MIT.

`docs/research/archived/` holds three relevant research artifacts:

- [architecture-review-high-load.md](./archived/architecture-review-high-load.md) — 42 KB, dated 2026-03-16. 17 findings for 100K+ agent deployments. Several already fixed in v1.5.3+ (writer lock, drop-oldest callback, reconnection initial state, etc.). Remaining pending: `QueueManager.Members` thread-safety, multi-server pool pattern, torn reads on Live entities. Only relevant if v1.12 commits to 100K+ scale.
- [2026-04-13-ari-exception-sites.md](./archived/2026-04-13-ari-exception-sites.md) — 16 KB. Inventory of 94 ARI call-sites that can return 404/409. **Already shipped in v1.6.0 Sprint 1 (task B1)**: `AriHttpExtensions.EnsureAriSuccessAsync(resource, id)` produces contextualized messages, 83/94 call-sites pass `(resource, id)`, `AriResourceErrorContextTests` covers it. The inventory was the PRE-fix audit; misread as pending during v1.12 planning. ~8 context-free call-sites (`CreateAsync`/`OriginateAsync`/etc.) can optionally be enriched with resource name without id in T4.1b.
- [2026-04-13-ami-gauge-audit.md](./archived/2026-04-13-ami-gauge-audit.md) — 10 KB. Coverage audit of Asterisk operational gauges; SDK covers 20 of 34 (59%). SLA / codec utilization / queue-target gauges are gaps. Nice-to-have, not strategic.

Cross-repo analysis of `Asterisk.Sdk.Pro` confirmed the MIT/Pro line is currently well drawn. A handful of Pro packages describe generic patterns (advanced routing, SignalR transport for Push, simplified Realtime DB config) that could reasonably be backported to MIT if the product strategy wanted to broaden the open-core surface — but none of them matches the strategic weight of what the external audit surfaced. The MIT/Pro boundary as currently drawn remains correct.

### External market audit

This is where the signal is loudest. Three converging data points:

**Asterisk 22/23 shipped a new Channel Driver that nobody in .NET has wrapped.** `chan_websocket` landed in Asterisk 22.8 / 23.2 (mid-2025) alongside `res_websocket_client`, exposing a binary-audio + JSON-control protocol that is the natural successor to AudioSocket for `ExternalMedia` scenarios. The protocol includes MEDIA_START, MEDIA_BUFFERING, MARK_MEDIA, SET_MEDIA_DIRECTION, XON/XOFF flow control, DTMF via TEXT frames. Asterisk also shipped **ARI Outbound WebSockets** (`application=outbound`) where Asterisk initiates the WS connection to the consumer app rather than accepting. Together these are the two biggest modernization vectors in the Asterisk ecosystem post-v22, and the .NET space is empty.

Fuentes: <https://docs.asterisk.org/Configuration/Channel-Drivers/WebSocket/> · <https://docs.asterisk.org/Configuration/Interfaces/Asterisk-REST-Interface-ARI/ARI-Outbound-Websockets/> · <https://www.asterisk.org/asterisk-media-over-websockets/>

**Voice AI provider landscape has shifted.** Three providers have high fit and high signal:

- **Cartesia Sonic-3** (TTS) achieves 40-90ms TTFA — the lowest in the market as of 2026. No official C# SDK. 15+ languages, emotion/laughter synthesis. Integrable via WebSocket + HTTP with our existing ADR-0014 "raw HTTP provider" pattern.
- **AssemblyAI Universal** (STT). The official AssemblyAI .NET SDK was discontinued in April 2025 (source: <https://www.assemblyai.com/blog/announcing-dotnet-sdk> announcement history). This is a vacuum we can fill with an AOT-clean client.
- **Speechmatics** (STT+TTS). Sub-150ms, 55+ languages, `$0.011 / 1K char` — roughly 27× cheaper than ElevenLabs. No C# SDK. Opens a price-sensitive enterprise segment.

**The multi-node gap has no .NET solution.** `CyCoreSystems/ari-proxy` (Go, NATS/RabbitMQ backend) is the de-facto solution for "pool of Asterisk nodes behind one ARI façade", and it has no .NET counterpart. Community forums routinely ask how to distribute ARI event processing across instances. A `Asterisk.Sdk.Push.Nats` bridge — publishing `RxPushEventBus` events to NATS subjects that mirror our topic hierarchy — is the minimum viable answer.

Fuentes: <https://github.com/CyCoreSystems/ari-proxy> · <https://cartesia.ai/sonic> · <https://www.speechmatics.com/>

Secondary findings:

- `asterisk-java` has been **dormant since October 2024** (v3.41.0). AsterNET has been **dormant since July 2018**. Official Asterisk Python/Node clients are flagged "best effort with limited support" with abandoned wrappers. The SDK ecosystem for Asterisk is badly attended; our competitive window is wide.
- Events added in asterisk-java post-v1.0 that we never ported: `ChannelTalkingStartEvent`/`ChannelTalkingStopEvent` (nov-2022), `BridgeVideoSourceUpdate` (sep-2024), `logintime` field on `QueueMemberEvent`. Trivial via our source generators.
- OpenTelemetry has **no semantic conventions for SIP/VoIP** (issue #2517 open). Our 9 ActivitySources + 13 Meters are already the most complete telemetry in the Asterisk SDK ecosystem; publishing a draft spec for SIP attributes (`sip.call_id`, `sip.response_code`, `asterisk.channel.id`, `call.direction`) positions us as reference.

## §4 What did not survive the audit

Several ideas that sound reasonable on paper did not earn a spot:

- **Webhook durable DLQ** was my initial recommendation before the audit. The investigation found no community demand for it, ADR-0011 explicitly declares the Push bus non-durable as a product choice, and Pro SDK is the right place for durability. Rejected.
- **PBX management dashboard / Call flow visualization** (two large parked proposals in `docs/research/archived/`). These are product work — they belong in the `Asterisk.Sdk.PbxAdmin` repo, which is exactly why they were separated to that repo's scope in 2026-03-24. Not for this SDK.
- **High-load fixes (`QueueManager` thread-safety, multi-server pool)** from `architecture-review-high-load.md`. Relevant only at 100K+ concurrent agent scale. No consumer is asking for that scale today. If the signal emerges, it is a v1.13+ scoping conversation.
- **Pro SDK backports (Routing, SignalR, Realtime).** The MIT/Pro boundary as drawn today is correct. Blurring it dilutes the Pro commercial offering without clear MIT benefit. Defer.
- **Asterisk 24 features.** GA expected January 2027. Premature.
- **Kafka / RabbitMQ brokers.** RabbitMQ's official .NET client is not AOT-clean. Kafka's `Confluent.Kafka` works but is heavyweight. NATS covers 80% of the broker-bridge use case with the best AOT story. Ship NATS first, evaluate others on demand.
- **Voice agent framework wrappers (Pipecat / LiveKit).** Better to provide primitives (media seam, VAD events, cancel-TTS action) that a consumer composes into any framework than to hard-couple to a moving target. Defer.

## §5 Strategic framing for v1.12.0

The investigation reframes v1.12.0 from "housekeeping patch" into "Asterisk 23 modernization + voice agent readiness". The narrative is coherent:

- The SDK declares itself the modern .NET SDK for Asterisk. Asterisk moved the goalposts in 22.8 / 23.2 with `chan_websocket` and Outbound WS; a modern SDK follows.
- The SDK declares itself voice-AI-ready since v1.9. In 2026 the market expanded past OpenAI+Azure+Google+ElevenLabs; Cartesia, AssemblyAI, Speechmatics are the three providers that matter most. A voice-AI-ready SDK covers them.
- The SDK declares itself operations-friendly since v1.7 with HealthChecks + ActivitySources. Multi-node deployments need a broker bridge; NATS is the minimum viable fit.
- The SDK declares itself disciplined (0 warnings, 0 trim warnings, 15 ADRs). The 9 remaining ADR candidates from the product alignment audit close the documentation loop.

Zero breaking changes. Package count grows 23 → 27 (four new packages: `Asterisk.Sdk.WebSocket`, `Asterisk.Sdk.VoiceAi.Cartesia`, `Asterisk.Sdk.VoiceAi.AssemblyAi`, `Asterisk.Sdk.VoiceAi.Speechmatics`, plus `Asterisk.Sdk.Push.Nats` makes five, bringing it to 28 — precise count to be confirmed in the active plan). Estimated 3-4 week release cycle.

## High-probability (ejecutables cuando el usuario lo pida)

### H1. Cross-repo consumer uptake — `Asterisk.Platform` bump 1.7.0 → 1.13.0
**What:** In the `Asterisk.Platform` repo, bump the SDK dependency from 1.7.0 to 1.13.0. Consume the new Push.Nats bidirectional bridge (if Platform runs multi-node), AsteriskSemanticConventions catalog (for consistent dashboards), AddEvent instrumentation (for chan_websocket DTMF/hangup/media traces).

**Why now:** Platform is ~6 versions behind. Every SDK release since 1.8 is untested against Platform's consumer surface; the longer the lag, the higher the chance a latent breaking change surfaces. Also blocks Platform from receiving field-validation data for OTel submission (see M1 below).

**Scope:** out-of-repo (Platform). Tracked here for coordination only.

**Effort:** 1 session (refresh NuGet ref, run Platform tests, address any cross-version drift).

### H2. VoiceAi provider investigation (market research, no code yet)
**What:** Evaluate three candidates for v1.14 VoiceAi additions: **ElevenLabs Flash 2.5** (TTS, claimed <150 ms TTFA — competitive with Cartesia Sonic-3), **OpenAI Whisper V3** (local-model STT, for air-gapped deployments), **Deepgram Aura 2** (TTS, already-integrated provider's new model).

**Why now:** VoiceAi expansion is the most visible MIT feature category for field adoption. Each candidate has different product positioning (Flash vs Cartesia = latency competition; Whisper V3 = air-gap niche; Aura 2 = no-new-provider convenience).

**Scope:** research doc in `docs/research/` with cost/quality/latency comparison, stakeholder decision, then if green-lit → ADR-0026 (raw HTTP/WebSocket pattern per ADR-0014) + provider subfolder + tests.

**Effort:** 1 research session (~2-4h) + 1-2 implementation sessions per approved provider.

## Medium-probability (esperando input externo)

### M1. OTel SIP conventions upstream submission
**What:** Contribute `AsteriskSemanticConventions` attribute set to `open-telemetry/semantic-conventions` (issue #2517 — open since 2022, no SIP profile yet). Submit the 54 consts across 11 nested classes as a draft proposal.

**Why now:** ADR-0025 §Notes requires ≥2 weeks production field validation before freezing upstream names. Once a real deployment runs v1.13 against prod traffic for 2+ weeks and the attribute set proves sufficient (no added tags, no renamed ones), open a comment on issue #2517 with the validated set.

**Blocker:** field validation deployment. Cannot start before at least one consumer reaches production with v1.13.

**Scope:** no code change — community submission + discussion thread + possibly attending OTel semantic-conventions WG meetings.

**Effort:** ~1 session to draft the contribution doc + ongoing community engagement (weeks to months).

### M2. Push.Backplane architectural decision (ADR, possibly ADR-0026)
**What:** Decide whether `IPushBackplane` abstraction over Push.Nats / Push.Webhooks / (Pro's) Push.Redis belongs in MIT. Three options:
- (a) **In MIT:** consumers register any backplane via DI; SDK owns the seam, Pro provides a Redis impl.
- (b) **In Pro:** Pro owns both the abstraction and its impls; MIT sticks to per-transport opt-in (current state).
- (c) **Neither:** leave it as three independent transports with no unified interface; encourage consumers to write their own glue.

**Why now:** With NATS bridge shipped bidirectional, the abstraction cost/benefit is clearer. Webhooks already has its own `IWebhookDelivery` seam; adding a backplane over that would be a net layer.

**Blocker:** Pro SDK roadmap has to decide whether it wants to own the abstraction. Depends on Pro telemetry sprint and whether Pro.Push is still in scope for the next Pro release.

**Scope:** ADR-0026 capturing the decision. If option (a), small refactor in MIT; if (b) or (c), doc-only.

**Effort:** 1 session for the ADR + decision discussion.

## Rejected (do not revisit without new driver)

- **Webhook durable DLQ** — ADR-0020 stands. Retry-only + dead_letter counter is the MIT shape. Durability is Pro territory.
- **JetStream durable consumers in Push.Nats** — ADR-0011 + ADR-0025 boundary. Pro provides the durable transport if needed.
- **`Asterisk.Sdk.Telemetry` consolidated package** — rejected v1.6 + v1.7 audits. AsteriskTelemetry helper + per-module sources/meters is sufficient.
- **Concrete-type event registry for Push.Nats round-trip** — rejected in ADR-0025. `RemotePushEvent` envelope is the MIT-level API; consumers needing stronger typing implement `INatsPayloadDeserializer` themselves.

## Default recommendation (if no new driver surfaces)

**Pause MIT feature work and pivot to Pro telemetry sprint.** v1.13 is a clean release; the highest-value next work is:

1. **Cross-repo bump** — H1 Platform 1.7.0 → 1.13.0 unlocks real consumer usage of v1.13 features, which in turn generates the field validation data M1 needs.
2. **Pro telemetry** — the Pro SDK has zero visibility into its own runtime today (no ActivitySources, no HealthChecks for its BackgroundServices). Closing that gap is higher-value per hour than any MIT addition currently on this list.
3. **Wait for drivers** — most M/L items are blocked-on-external-input. Patience is correct.

If the user has a concrete product goal that changes this calculation (e.g., "we need a Kafka bridge for partner X"), this document should be revised before starting new work.

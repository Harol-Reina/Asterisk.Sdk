# Plan: v1.12.0 scope ‚Äî Asterisk 23 modernization + voice agent readiness

**Date:** 2026-04-19
**Target release:** v1.12.0 (estimated 3-4 weeks from start)
**Research basis:** [docs/research/2026-04-19-v1.12.0-product-opportunities.md](../../research/2026-04-19-v1.12.0-product-opportunities.md)

---

## Context

v1.11.1 shipped 2026-04-19 with a full product alignment audit that identified the SDK as internally coherent, API-frozen, and operationally mature. Following user instruction to evaluate v1.12 "con foco en el producto final, sin atajos", a three-angle investigation (internal codebase, deferred work archive, external market) was run. The investigation converged on a strategic scope different from the housekeeping release initially proposed.

The consensus: Asterisk shipped new infrastructure in 22.8 / 23.2 (`chan_websocket`, ARI Outbound WebSocket) that no .NET SDK covers, the voice-AI provider landscape moved past our 2025 provider set (Cartesia, AssemblyAI, Speechmatics are the new frontier), and the multi-node deployment gap has no .NET bridge. These three vectors define v1.12 as an **Asterisk 23 modernization + voice-agent-ready release**, not a documentation pass.

Zero breaking changes. Package count grows from 23 to 28 (five new packages). Estimated 3-4 weeks.

---

## Scope ‚Äî four tiers

### Tier 1 ‚Äî Strategic core (Asterisk 23 alignment)

These two items are load-bearing for the narrative "this is the modern .NET Asterisk SDK".

**1.1 Extend `WebSocketAudioSession` with `chan_websocket` JSON control protocol** ‚Äî **scope narrowed 2026-04-19.** The archived plan [../completed/plan-chan-websocket-integration.md](../completed/plan-chan-websocket-integration.md) was executed in an earlier sprint: `WebSocketAudioServer` / `WebSocketAudioSession` / `AudioSocketServer` / `CompositeAudioServer` / `AudioChannelVars` already exist under `src/Asterisk.Sdk.Ari/Audio/`. The gap is narrow: the session handles BINARY audio frames but ignores TEXT frames. Asterisk 22.8 / 23.2+ sends control messages (MEDIA_START, MEDIA_BUFFERING, MARK_MEDIA, SET_MEDIA_DIRECTION, XON/XOFF, DTMF) as JSON over TEXT frames.

Work to ship in v1.12:
- Add `ChanWebSocketControlMessage` types (sealed records) for each message in the protocol.
- Add `ChanWebSocketJsonContext` with `[JsonSerializable]` for AOT parsing/emitting.
- Extend `WebSocketAudioSession.ReadPumpAsync` to dispatch TEXT frames to the control parser and emit via a new `IObservable<ChanWebSocketControlMessage>`.
- Add send-side methods: `SendMarkAsync(mark)`, `SendXonAsync()`, `SendXoffAsync()`, `SendSetMediaDirectionAsync(direction)`.
- Unit tests: JSON round-trip for each message type, TEXT frame dispatch in session.
- Functional test: real Asterisk 23.2+ container with `chan_websocket` configured, verify control messages flow.
- NO new package ‚Äî extensions live in `src/Asterisk.Sdk.Ari/Audio/` alongside existing server.

**1.2 ARI Outbound WebSocket support** ‚Äî extend `Asterisk.Sdk.Ari` to act as the WS server that Asterisk initiates the connection to (`application=outbound` in `ari.conf` + `res_websocket_client`). New `IAriOutboundListener` + DI registration. Handle `ApplicationRegistered` and `ApplicationUnregistered` events. Persistent and per-call modes.

- Unit tests: handshake, registration events, disconnect handling.
- Functional tests: Asterisk 22.5+ container configured with outbound WS.
- Example: `Examples/AriOutboundExample/`.

### Tier 2 ‚Äî VoiceAi provider expansion

Three new provider packages, following the ADR-0014 raw HTTP / ClientWebSocket pattern. Each ~200-300 LOC + tests.

**2.1 `Asterisk.Sdk.VoiceAi.Cartesia`** ‚Äî STT (Ink-Whisper, optimized for conversational telephony) + TTS (Sonic-3, 40-90ms TTFA). WebSocket streaming for both. No official C# SDK, so zero vendor SDK dependency.

**2.2 `Asterisk.Sdk.VoiceAi.AssemblyAi`** ‚Äî STT Universal streaming endpoint via WebSocket v3. Fills the vacuum left by the discontinued official .NET SDK (April 2025).

**2.3 `Asterisk.Sdk.VoiceAi.Speechmatics`** ‚Äî STT + TTS. Sub-150ms, 55+ languages, 27√ó cheaper than ElevenLabs. Opens enterprise price-sensitive segment.

Each provider package:
- Implements `SpeechRecognizer` or `SpeechSynthesizer` from `Asterisk.Sdk.VoiceAi.Stt` / `.Tts`.
- Overrides `ProviderName` as a `const string` (ADR-0005 compliance).
- Ships its own `*Options.cs` with `[OptionsValidator]`.
- Adds coverage to `VoiceAiBenchmarks` with TTFA / TTFB latency measurements.
- Adds integration example under `Examples/`.

### Tier 3 ‚Äî Distribution and operations

**3.1 `Asterisk.Sdk.Push.Nats` ‚Äî new package** ‚Äî NATS bridge for `RxPushEventBus`. Publishes every push event to a configurable NATS subject hierarchy that mirrors the topic hierarchy (ADR-0007 semantics: `**`, `{self}`, `*` translate to NATS subject tokens). Uses `NATS.Net` v2 (AOT-aware). Enables multi-node deployment: one NATS cluster, N SDK instances, each publishes local events and optionally subscribes to remote events with topic-based filtering.

- Integration test: Testcontainers NATS image, publish from one SDK instance, verify subscription from another.
- Example: `Examples/NatsBridgeExample/` showing a two-node split.

**3.2 `.github/workflows/publish.yml`** ‚Äî GitHub Actions workflow triggered on `v*` tag push. Uses `NUGET_API_KEY` GitHub secret. Publishes all 28 packages to nuget.org automatically. Closes the concrete risk documented in v1.11.1 release notes (key exposure during manual publish). Keeps the CLAUDE.md promise that was previously unfulfilled.

### Tier 4 ‚Äî Polish and documentation

**4.1 ARI exception context mapping** ‚Äî **ALREADY SHIPPED in v1.6.0 Sprint 1 (task B1).** `AriHttpExtensions.EnsureAriSuccessAsync(resource, id)` already exists with `BuildSubject`/`Suffix`/`Capitalize` helpers producing messages like `"Channel 'SIP/1234-00000001' not found: <body>"`. 83 of 94 call-sites already pass `(resource, id)`; the ~11 remaining are genuinely context-free (List/Ping/GetInfo/Originate). Tests in `Tests/Asterisk.Sdk.Ari.Tests/Resources/AriResourceErrorContextTests.cs` cover it. The archived research doc `2026-04-13-ari-exception-sites.md` was the PRE-fix inventory; misread as pending during v1.12 planning. No work required here.

**4.1b (optional, ~30 min)** ‚Äî enrich the ~8 call-sites that still call `EnsureAriSuccessAsync()` parameterless but could add resource context without id: `AriChannelsResource.CreateAsync/OriginateAsync/CreateExternalMediaAsync/CreateWithoutDialAsync` (pass `"channel"`), `AriBridgesResource.CreateAsync` (pass `"bridge"`), `AriClient.GenerateUserEventAsync` (pass `"event", eventName`), `AriEndpointsResource.SendMessageAsync`. Optional polish; non-blocking for release.

**4.2 New AMI events (Asterisk 22/23 gaps)** ‚Äî **ALREADY SHIPPED pre-v1.12.** All six items verified present in the codebase:
- `ChannelTalkingStartEvent` at `src/Asterisk.Sdk.Ami/Events/ChannelTalkingStartEvent.cs` with `[AsteriskMapping("ChannelTalkingStart")]`, inherits `ChannelEventBase`. PublicAPI.Shipped.txt line 2001.
- `ChannelTalkingStopEvent` at same folder with `Duration` field. PublicAPI.Shipped.txt line 2003.
- `BridgeVideoSourceUpdateEvent` with `BridgePreviousVideoSource` field. PublicAPI.Shipped.txt line 1843.
- `ApplicationRegisteredEvent` / `ApplicationUnregisteredEvent` (ARI events, not AMI) in `src/Asterisk.Sdk.Ari/Events/AriEvents.cs:309,315`.
- `Logintime` field on `QueueMemberEvent.cs:25` (type `int?`).

The external product-opportunities research (¬ß2 "secondary findings") flagged these as absent based on asterisk-java comparison, but the comparison was stale ‚Äî these landed in an earlier release cycle. Removed from v1.12 scope.

**4.3 OpenTelemetry SIP semantic conventions draft** ‚Äî new research doc at `docs/research/2026-04-XX-otel-sip-semantic-conventions.md` proposing attribute names for `sip.call_id`, `sip.response_code`, `asterisk.channel.id`, `asterisk.bridge.id`, `call.direction`, `call.state`. Align our 9 ActivitySources to emit these attributes. No behavior change, just naming. Positions the SDK as reference implementation for SIP/VoIP OTel.

**4.4 The 9 deferred ADRs** ‚Äî write in three batches as described in the product alignment audit ¬ß4 and in the v1.11.1 release notes:
- VoiceAi batch: `ProviderName` override (ADR-0016), AudioSocket codec negotiation (ADR-0017).
- Sessions + Push batch: reconciliation strategy (ADR-0018), `TraceContext` ambient capture (ADR-0019), webhook no-DLQ (ADR-0020).
- Repo policy batch: heartbeat strategy (ADR-0021), Activity cancellation semantics (ADR-0022), PublicAPI tracker policy (ADR-0023), `BannedSymbols.txt` as AOT policy (ADR-0024).

Bringing the ADR count from 15 to 24. Each ADR is pure documentation, no code change.

## Out of scope (explicit)

- Webhook durable DLQ ‚Äî no demand signal, conflicts with ADR-0011.
- PBX management dashboard / call flow visualization ‚Äî belongs to `Asterisk.Sdk.PbxAdmin` repo.
- High-load fixes for 100K+ agents ‚Äî defer to v1.13+ when a consumer surfaces the need.
- Pro SDK feature backports (Routing, SignalR, Realtime) ‚Äî preserve MIT/Pro boundary.
- Asterisk 24 features ‚Äî GA expected January 2027, premature.
- Kafka / RabbitMQ bridges ‚Äî NATS first; evaluate others when demand emerges.
- Voice agent framework wrappers (Pipecat / LiveKit) ‚Äî ship primitives, not wrappers.

## Risk register

- **`chan_websocket` protocol evolves.** Mitigation: target Asterisk 23.3 as the baseline, document minimum Asterisk version in README, add compatibility notes to the package README.
- **VoiceAi provider APIs change.** Mitigation: every provider has its own `*Options` + integration test against the live API; if a vendor breaks compat, we ship a point release for that package alone. This is the reason ADR-0014 exists.
- **NATS.Net v2 AOT gaps.** Mitigation: validate with `PublishAot=true` on `Examples/NatsBridgeExample/` as part of acceptance. If blocked, scope the package down to a provider-agnostic `IPushEventForwarder` interface and ship NATS as a separate follow-up.
- **4-week estimate slips to 6 weeks.** Mitigation: tiers are independent; if time pressure mounts, drop Tier 4.4 (ADRs) to v1.12.1, ship the code-carrying tiers 1-3 + 4.1-4.3 as v1.12.0.
- **Manual NUGET_API_KEY exposure risk before publish.yml lands.** Mitigation: T3.2 is Week 1 Task 2. Any pre-v1.12 hotfix release in the interim continues to require manual publish ‚Äî this is the existing risk being mitigated, not a new one.

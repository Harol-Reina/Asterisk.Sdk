# v0.5.0-beta Design Spec

**Date:** 2026-03-17
**Status:** Approved
**Scope:** Activities stabilization, Session Engine examples, functional tests, release

## Overview

v0.5.0-beta is the last MIT release before PRO Foundation (v0.1.0-pro-beta). Its goal is to stabilize the MIT API surface so PRO can build on top without risk. The approach is: fix bugs → validate with examples → test what's validated → release.

## Approach

**Enfoque C: Activities fix primero, luego ejemplos como validación, tests al final**

```
Sprint 1: Activities Fixes → Sprint 2: Ejemplos → Sprint 3: Tests funcionales → Sprint 4: Release
```

Each sprint feeds the next without rework.

## Sprint 1 — Activities Fixes

### 1.1 Verify and harden `ActivityBase.CancelAsync()`

**Status:** Cancellation was already wired in Phase 1 Sprint 2 — `_executionCts.Token` is passed to `ExecuteAsync` and `CancelAsync()` calls `_executionCts?.Cancel()`.

**Minor fix:** In `CancelAsync()`, `_executionCts?.Cancel()` is called AFTER `SetStatus(Cancelled)`. This means the status changes before execution actually stops. Swap the order: cancel first, then set status — so observers see the status change only after the operation is truly cancelled.

**Verification:** Add tests to confirm the existing mechanism works end-to-end:
- `CancelAsync_ShouldCancelRunningExecuteAsync` — verify real CTS propagation
- `CancelAsync_ShouldThrow_WhenNotStarted`
- `CancelAsync_ShouldBeIdempotent_WhenCalledTwice`

### 1.2 Fix `BlindTransferActivity`

**Problem:** Uses `Goto()` (dialplan jump) instead of `BlindTransfer()` (real SIP transfer).

**Fix:**
- Change implementation to use AGI command `Exec("BlindTransfer", $"{Destination.Extension}@{Destination.Context}")`
- Keep public API identical (`required DialPlanExtension Destination`)
- Add test validating `BlindTransfer` is invoked, not `Goto`

### 1.3 Refactor `ExternalMediaActivity`

**Problem:** Does not inherit from `ActivityBase`, has 50ms polling loop with `Task.Delay`, no `Lock`.

**Architectural note:** `ActivityBase` requires `IAgiChannel` (AGI protocol) but `ExternalMediaActivity` uses `IAriClient` (ARI protocol). These are fundamentally different APIs. Making it inherit from `ActivityBase` directly is not possible.

**Fix:**
- Create `AriActivityBase` — a parallel base class for ARI-based activities, following the same `IActivity` contract and state machine pattern as `ActivityBase` but accepting `IAriClient` instead of `IAgiChannel`
- Migrate `ExternalMediaActivity` to inherit from `AriActivityBase`
- Replace polling loop with `TaskCompletionSource` that completes when the audio server receives a connection
- Add `Lock` for thread-safe status transitions (matching `ActivityBase` pattern)
- Remove duplicate `BehaviorSubject` (provided by `AriActivityBase`)
- Keep public API: `App`, `ExternalHost`, `Format`, `AudioStream`, `Channel`

### 1.4 Eliminate `CallStateData` dead code

**Problem:** 4 classes (`CallStateData`, `CallStateAnswered`, `CallStateNewInbound`, `CallStateParked`, `CallStateTransfer`) defined but never used anywhere.

**Fix:** Delete the files. If needed in the future, redesign with Sessions context.

### 1.5 Remove `[Experimental("ASTSDK001")]`

After all fixes above, remove the attribute from `IActivity` and the `NoWarn` suppression from `.csproj`.

## Sprint 2 — Examples as Validation

### 2.1 `SessionExample` (basic console)

**Purpose:** An evaluator connects, sees sessions in real-time, understands the value in 2 minutes.

**Structure:**
```
Examples/SessionExample/
├── Program.cs
└── SessionExample.csproj
```

**Flow:**
1. Configure `AddAsterisk()` + `AddAsteriskSessions()` via `HostApplicationBuilder`
2. Connect to Asterisk AMI
3. Subscribe to `ICallSessionManager.SessionEvents` (observable `SessionDomainEvent`)
4. Print to console with colors:
   - `[NEW]` Session created (green)
   - `[RING]` Ringing (yellow)
   - `[CONN]` Connected — show participants (cyan)
   - `[HOLD]` On hold (magenta)
   - `[END]` Completed — show duration, talk time (red)
5. Every 10s print summary: active, completed, failed sessions
6. Ctrl+C to exit

**Validates:**
- DI registration ergonomics (`AddAsteriskSessions()`)
- That `SessionDomainEvent` has useful information for consumers
- That the observable works correctly for external subscribers

### 2.2 `SessionExtensionsExample` (extension points)

**Purpose:** An implementor/PRO evaluator sees how to customize the Session Engine.

**Structure:**
```
Examples/SessionExtensionsExample/
├── Program.cs
├── FileSessionStore.cs
└── SessionExtensionsExample.csproj
```

**Content:**

**`FileSessionStore`** — Custom `SessionStoreBase` implementation:
- Override `SaveAsync` → write/update JSON to file (one line per session, upsert by sessionId)
- Override `GetAsync` → read session by ID from file
- Override `GetActiveAsync` → return all non-completed sessions from file
- Override `DeleteAsync` → remove session line from file
- Use `System.Text.Json` source-generated for AOT

**`Program.cs`:**
1. Register `FileSessionStore` as replacement for default `InMemorySessionStore`
2. Connect to Asterisk
3. Generate calls → sessions persist to file
4. On exit, show file content as proof

**Prerequisite:** `CallSessionManager` currently uses internal `ConcurrentDictionary` fields directly and does NOT delegate to `SessionStoreBase`. As part of this sprint, wire `CallSessionManager` to inject and use `SessionStoreBase` for session persistence. This enables DI replacement of the default `InMemorySessionStore`.

**Validates:**
- `SessionStoreBase` has the correct methods for a real store
- `CallSessionManager` correctly delegates to `SessionStoreBase`
- DI allows replacing default implementations
- JSON serialization works with AOT
- Extension points general ergonomics

### 2.3 Fix `PbxActivitiesExample`

**Problem:** Existing example doesn't demonstrate real Activities — shows Live API instead.

**Fix:** Rewrite to demonstrate real usage of `DialActivity`, `HoldActivity`, `QueueActivity` on an `IAgiChannel`. Consistent with Sprint 1 fixes.

## Sprint 3 — Functional Tests

### 3.1 Multi-layer in-memory tests (~30-40 tests)

**Project:** `Tests/Asterisk.Sdk.Sessions.FunctionalTests/`

**Strategy:** Feed simulated AMI events through the real pipeline `AmiConnection → Live → SessionManager` without granular mocks. Only substitute TCP transport with a `FakeAmiTransport` that injects events.

**Prerequisite:** `FakeAmiTransport` must be built as part of this sprint. It implements the transport abstraction (`PipelineSocketConnection` or equivalent) to inject raw AMI event strings into the `AmiConnection` pipeline without a real TCP connection. This is a non-trivial component (~100-150 lines) that requires understanding `System.IO.Pipelines` and the AMI protocol framing.

#### Session Lifecycle (8-10 tests)
- Inbound call: `Newchannel → Ringing → Answer → Hangup` → session goes `Created → Ringing → Connected → Completed`
- Outbound call: `Newchannel(Local) → Dial → Answer → Hangup` → same progression
- Failed call: `Newchannel → Hangup(cause=21)` → session `Created → Failed` with cause in metadata
- Call with timeout: `Newchannel → Dial` without Answer → reconciler marks `TimedOut`
- Transfer: `Connected → BlindTransfer event` → session `Connected → Transferring → Connected` with new participant
- Hold/Unhold: `Connected → Hold → Unhold` → session computes `HoldTime` correctly
- Conference: `Connected → ConfbridgeJoin` → session in `Conference` with N participants
- Duration: verify `Duration`, `WaitTime`, `TalkTime` are coherent

#### LinkedId Correlation (6-8 tests)
- Two channels with same LinkedId → same session
- Channel without LinkedId → individual session
- Empty LinkedId → fallback to UniqueId
- 3 channels (caller + agent + Local bridge) → one session with 3 participants
- Channel joins existing session mid-call → participant added correctly
- Two simultaneous calls with different LinkedIds → two separate sessions

#### Reconciliation and edge cases (6-8 tests)
- Orphan detection: session with channels but all hangup without event → reconciler marks `Failed`
- Configurable timeout: `SessionOptions.DialingTimeout = 5s` → respected
- Reconciler does not touch active sessions with live channels
- Session in `Completed` not modified by reconciler
- Burst of 50 simultaneous events → all indices consistent (thread safety)
- AMI reconnection → sessions cleaned correctly

#### Indices and queries (5-6 tests)
- `GetByChannelId` returns correct session
- `GetByLinkedId` returns correct session
- `GetActiveSessions` excludes completed/failed
- `SessionStore.GetByIdAsync` after multiple updates has fresh data
- Bridge ID lookup works with `_bridgeToSession` index

#### Domain events (5-6 tests)
- `CallStarted` emitted on session creation
- `CallConnected` emitted on Answer
- `CallEnded` emitted on completion
- Subscriber receives events in correct order
- Multiple subscribers receive the same event

## Design Decisions

### Why fix Activities before examples
Activities has real bugs (CancelAsync, BlindTransfer). Fixing first means examples consume stable APIs. Writing tests last means they're written once against the final API.

### Why two separate examples instead of one combined
The 10 existing examples each do one thing well. A combined example dilutes both messages and breaks the project pattern. Two focused examples serve two distinct audiences (evaluator vs implementor).

### Why 80/20 split on in-memory vs Docker tests
Critical bugs (orphan leaks, race conditions, timeout edge cases) are only testable with deterministic in-memory tests. Docker tests validate happy paths against real Asterisk. The bugs that cause 3am pages are the edge cases, not the happy paths.

### Why no Activities ↔ Sessions integration in v0.5.0-beta
Both layers are complementary but independent: Sessions observes (AMI events), Activities executes (AGI commands). Integrating without real usage feedback risks freezing a wrong design. Defer to v0.6.0 with data.
